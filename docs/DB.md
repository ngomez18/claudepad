# Database

## Location

```
~/.claudepad/claudepad.db
```

The directory is created with `0700` permissions on first launch if it does not exist. The database file is created by SQLite on first open.

## Purpose

SQLite stores enrichment metadata only. The `.claude/` files on disk are always the source of truth — Claudepad never renames or deletes them. The DB layer adds friendly names, tags, notes, a project registry, and usage snapshots on top of what already exists on the filesystem.

## Driver

`modernc.org/sqlite` — pure Go, no CGo. Required for Wails cross-compilation. Registers under the driver name `"sqlite"` (not `"sqlite3"`).

## Connection settings

| Setting | Value | Reason |
|---------|-------|--------|
| Journal mode | WAL | Better read/write concurrency |
| Foreign keys | ON | Referential integrity |
| Busy timeout | 5000 ms | Avoids immediate `SQLITE_BUSY` errors |
| Max open conns | 1 | Single-user app; prevents locking issues with WAL |

DSN: `<path>?_journal_mode=WAL&_foreign_keys=on&_busy_timeout=5000`

## Schema

### `projects`

Tracks Claude Code project directories known to Claudepad.

| Column | Type | Notes |
|--------|------|-------|
| `id` | TEXT PK | UUID |
| `name` | TEXT | User-assigned friendly name |
| `real_path` | TEXT UNIQUE NOT NULL | Absolute path to project root |
| `is_global` | INTEGER | 1 = global `~/.claude` entry |
| `last_opened` | DATETIME | |
| `created_at` | DATETIME | Defaults to `datetime('now')` |

### `file_metadata`

Enrichment for individual files managed by Claudepad (plans, hooks, skills, commands, etc.).

| Column | Type | Notes |
|--------|------|-------|
| `id` | TEXT PK | UUID |
| `real_path` | TEXT UNIQUE NOT NULL | Absolute path to the file |
| `file_type` | TEXT NOT NULL | e.g. `plan`, `hook`, `skill`, `command` |
| `friendly_name` | TEXT | User-assigned display name |
| `tags` | TEXT | JSON array, e.g. `["work", "debug"]` |
| `notes` | TEXT | Free-form markdown notes |
| `archived` | INTEGER | 1 = hidden from default views |
| `created_at` | DATETIME | |
| `updated_at` | DATETIME | |

### `usage_snapshots`

Point-in-time copies of `stats-cache.json` for historical trending.

| Column | Type | Notes |
|--------|------|-------|
| `id` | INTEGER PK AUTOINCREMENT | |
| `snapshot_date` | DATE UNIQUE NOT NULL | One row per calendar day |
| `raw_json` | TEXT NOT NULL | Full JSON blob from `stats-cache.json` |
| `captured_at` | DATETIME | |

### `app_settings`

Key/value store for application preferences.

| Column | Type | Notes |
|--------|------|-------|
| `key` | TEXT PK | Setting identifier |
| `value` | TEXT NOT NULL | JSON-encoded value |
| `updated_at` | DATETIME | |

## Migrations

Migrations are managed by [goose v3](https://github.com/pressly/goose) using embedded SQL files.

**Location:** `backend/db/migrations/`

**Format:** Sequential timestamp prefix generated by the goose CLI — never write migration files by hand.

### Adding a migration

```bash
go run github.com/pressly/goose/v3/cmd/goose@latest \
  -dir backend/db/migrations create <name> sql
```

This generates `backend/db/migrations/<timestamp>_<name>.sql` with `-- +goose Up` / `-- +goose Down` stubs. Fill in the SQL, then commit both the migration file and any updated `go.sum`.

### How migrations run

`db.Open()` calls `provider.Up()` on every app startup. Goose tracks applied versions in the `goose_db_version` table and is a no-op when the schema is already current. Migrations run automatically — no manual step required.

### Rules

- Never edit or delete an existing migration file after it has been committed.
- Always provide a `-- +goose Down` block so migrations can be rolled back during development.
- Migrations must be idempotent where practical (`CREATE TABLE IF NOT EXISTS`, etc.).
